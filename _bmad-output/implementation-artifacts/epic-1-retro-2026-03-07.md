## Challenges & Growth Areas

### Database Configuration (Story 1.3)
- **Issue:** Original plan was SQLite for local development, Postgres for production
- **Problem:** Payload migrations are incompatible between database types
- **Resolution:** Pivoted to "Postgres everywhere" - same database type for local and production
- **Impact:** Multiple deployment iterations (5+ fix commits), Architecture doc updated
- **Lesson:** Test database migration compatibility before committing to split-database approach

### Observability Integration (Story 1.4)
- **Issue:** Planned approach (`@vercel/otel` with `registerOTel`) didn't work well with Axiom in serverless
- **Problem:** Poor debugging feedback loop - deploy, wait, check Axiom, no traces, no idea why
- **Resolution:** Used direct OpenTelemetry SDK with custom configuration, `SimpleSpanProcessor` for immediate export
- **Impact:** 15 fix commits, significant debugging time
- **Lesson:** Spike integration-heavy work before full implementation

### Implementation vs. Plan Divergence
- **Pattern:** Planned approaches didn't match reality for infrastructure integrations
- **Lesson:** Architecture documentation should be treated as living document, updated when reality differs

---

## Key Insights

1. **Spike integration-heavy work first** - A 30-minute proof-of-concept could have saved hours of debugging on Axiom/OpenTelemetry
2. **Architecture docs are living documents** - Update them when implementation diverges from plan
3. **Code reviews are working** - Continue them; they caught real issues in every story
4. **Serverless has specific constraints** - BatchSpanProcessor doesn't work (function terminates before batch ships), need SimpleSpanProcessor
5. **Infrastructure-first paid off** - Security, observability, and CI/CD in place from day one
