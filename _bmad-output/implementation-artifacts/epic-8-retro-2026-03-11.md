## What Didn't Go Well

### Discord Workflow Issues (Story 8.1)
- Workflow failed on Dependabot PRs due to:
  - Dependabot secrets being separate from Actions secrets
  - Complex PR bodies breaking JSON construction
  - Concurrency groups cancelling parallel notifications
- Story was marked "done" without real-world verification

### Dependabot Configuration Gaps (Story 8.2)
- No grouping configured - resulted in 9 separate PRs
- Security alerts (UI) vs version updates (yml) confusion
- 4 security vulnerabilities detected but no auto-fix PRs created

### Verification Gap
- Stories with "Manual Verification Required" sections weren't actually verified
- Code complete was treated as feature complete
- External integrations need real triggers to validate

---

## Issues Fixed During Retrospective

| Issue | Root Cause | Fix |
|-------|-----------|-----|
| Discord notifications failing on Dependabot PRs | Dependabot can't access Actions secrets | Added secret to Dependabot secrets store |
| JSON 400 errors from Discord | Complex PR bodies breaking shell variable escaping | Simplified workflow using proven pattern |
| Notifications cancelling each other | Shared concurrency group across all PRs | Changed to per-PR concurrency groups |
| 9 separate Dependabot PRs | No grouping configured | Added groups for @opentelemetry/*, @vercel/*, actions, other |

### Commits Applied During Retro
1. `fix(ci): handle complex PR bodies in Discord notifications`
2. `fix(ci): use per-PR concurrency group for Discord notifications`
3. `fix(ci): simplify Discord notification workflow`
4. `chore(deps): add grouping to reduce Dependabot PR noise`

---

## Key Insights

1. **External integrations need real-world testing** - Code review and "looks right" isn't enough for webhooks, GitHub Actions, and third-party services
2. **Dependabot has two separate systems** - Version updates (yml) and security updates (UI) don't automatically coordinate
3. **Stories marked done weren't production-ready** - Manual verification steps were skipped
4. **Live debugging in retro was valuable** - Found and fixed real issues that would have caused ongoing problems
