## Challenges & Growth Areas

### Epic Over-Decomposition

**Issue:** Story 6.2 (OG/Twitter cards) was absorbed into Story 6.1 because the Next.js Metadata API handles meta tags, OG tags, and Twitter cards in a single configuration object. Splitting them into separate stories created artificial boundaries.

**Pattern:** This mirrors Epic 4 where Stories 4.3 and 4.4 were primarily verification of earlier work. Two consecutive epics have shown story boundaries that don't match natural implementation boundaries.

**Recommendation:** In future planning, consider the implementation unit when decomposing stories. If two stories modify the same file/object, they may belong together.

### Accessibility Boilerplate in Dev Notes

**Issue:** Every story file included the same "Accessibility Patterns from Epic 2 Retrospective" boilerplate - color system, semantic HTML, React keys, focus states - regardless of relevance. Story 6.5 (a static text file) had the full block.

**Resolution:** Remove boilerplate from story template. Create a dev standards reference in `CLAUDE.md` that story creators can pull from when relevant.

### Local Build Command Hangs

**Issue:** `pnpm build` runs `payload migrate && next build`, requiring a database connection. Locally, this hangs indefinitely without a running database, wasting time when agents attempt build verification.

**Resolution:** Create `CLAUDE.md` documenting that local verification should use `npx next build` directly. The `payload migrate` step is required for Vercel deployments and must remain in the build command.

### No Test Framework

**Issue:** No unit or integration test framework is configured. Verification relies entirely on build checks and CI Lighthouse runs. This has been true since Epic 1.

**Status:** Tracked as Story 8.5 (Playwright E2E tests) in Epic 8. No separate action item needed, but flag if deprioritized beyond Epic 8.

## Key Insights

1. **Built-in framework features reduce maintenance burden** - Using Next.js native capabilities for all SEO features means zero dependency management overhead
2. **Story decomposition should follow implementation boundaries** - Two epics have shown that artificial story splits create tracking overhead without value
3. **Template boilerplate needs relevance filtering** - Shared guidance should live in a central reference, not be copy-pasted into every story
4. **Build commands may need environment-specific variants** - What works for deployment (migrations) doesn't work for local verification
